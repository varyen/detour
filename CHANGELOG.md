# Changelog

Все заметные изменения проекта Detour.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/),
версионирование — [SemVer](https://semver.org/lang/ru/).

## [1.23.0] — 2026-07-03

Новое: авто-восстановление аппаратного ускорения (offload) на QCA-роутерах. На home
(GL-BE9300, ipq53xx) время от времени скорость LAN↔WAN падала до ~100 Мбит и лечилась
только перезагрузкой. Причина — «залипание» аппаратного ускорителя NSS/PPE: форвардинг
сваливается на CPU. Живой A/B-тест подтвердил: здоровый форвардинг идёт мимо CPU
(softirq низкий даже на 330 Мбит), а при залипании — всплеск softirq при простое прокси.

### Добавлено

- **Watchdog + авто-восстановление offload (`detour-offload`, cron раз в минуту).** Ловит
  залипание по «отпечатку» — устойчивый NET softirq при незанятом userspace-прокси
  (проксирование грузит user-CPU, а не softirq, поэтому не путается) — и восстанавливает
  ускоритель **без перезагрузки**: лёгкий сброс ECM-фронтендов (только запущенных;
  `gl_front_end_*` в состоянии «stop» не трогает) → при неуспехе `qca-nss-ecm restart`.
  Порог + стрик из 2 тиков + кулдаун защищают от ложных срабатываний. Результат уходит
  Web Push. _OpenWrt/QCA (ipq53xx)._
- **Панель → меню → «Ускорение (offload)»**: выбор режима (авто-восстановление / только
  уведомлять / выкл), кнопка «Пнуть ускоритель сейчас», статус (режим, последнее действие,
  softirq). Конфиг `/etc/detour/offload.conf` (по умолчанию `auto`; переживает апгрейд
  через keeplist — выбор пользователя не сбрасывается).

### Только QCA / Keenetic

- Фича **только для QCA-NSS** (ipq53xx, напр. GL-BE9300) — как NFQUEUE, физически привязана
  к железу. На Keenetic (mipsel) и MediaTek (MT6000) такого ускорителя нет: `detour-offload`
  не входит в их сборку, а в UI пункт скрывается по `offload_status.supported`. Честный
  паритет: применить физически нельзя, поэтому задизейблено, а не «сломано».

### Важно (честно)

- Безопасность «kick» подтверждена на живом home (лёгкий сброс не рвёт соединения,
  `gl_front_end_*` сохраняются). Но **эффективность на реальном залипании ещё не проверена**
  — воспроизвести залипание по команде нельзя; watchdog зафиксирует «до/после», когда это
  случится. Крайняя мера, если kick не поможет, — перезагрузка роутера.

## [1.22.3] — 2026-07-03

Восстановление HTTPS-доступа к панели и защита от повторения. У `detour-cert`
(nginx-режим) была дыра: сертификат ставился, но маршрут HTTPS `<домен>→панель`
оставался «на совести пользователя» — из-за чего валидный Let's Encrypt-серт мог
молча подменяться дефолтным сертификатом роутера, и панель становилась недоступна
по HTTPS (симптом: браузер ругается на несовпадение имени сертификата).

### Исправлено

- **Панель снова доступна по HTTPS, даже если у домена нет своего `server`-блока в
  nginx.** Раньше `detour-cert` в nginx-режиме только клал файлы серта в
  `/etc/nginx/ssl/<домен>/` и рассчитывал, что маршрут пропишет сам пользователь.
  Если именованного `server`-блока не было (например, его затёрло обновление
  прошивки GL.iNet, перегенерившее `gl.conf`), запрос по SNI `<домен>` попадал в
  дефолтный :443-сервер роутера и отдавал **чужой** самоподписанный серт
  (`console.gl-inet.com`) — при живом и валидном LE-серте. Теперь `detour-cert
  issue` дополнительно ставит управляемый именованный vhost
  `zz-detour-panel-tls.conf` (префикс `zz-` → грузится последним, дефолтный сервер
  роутера и доступ по IP не трогаются; блок выигрывает по SNI), проксирующий домен
  на панель (порт панели берётся из `uhttpd.main.listen_http`, дефолт `:8080`).
  Пишется только если домен ещё не отдаёт свой серт, под `nginx -t` с откатом.
  _OpenWrt/GL.iNet (nginx)._
- **Панель подсвечивает, когда HTTPS отдаёт «не тот» сертификат.** `detour-cert
  detect` теперь делает живую проверку — какой серт реально отдаёт локальный :443
  по SNI домена (`serving_ok`/`serving_cn`). Модалка сертификата показывает
  расхождение красным и подсказывает перевыпустить (это переустановит маршрутный
  vhost). Пустой ответ (нет `openssl s_client`) трактуется как «неизвестно», а не
  как ошибка — ложных предупреждений нет.

### Keenetic

- Паритет соблюдён без отдельного кода: на Keenetic панель обслуживает **lighttpd**,
  который сам слушает :443 с сертом (`configure_lighttpd`), — класс бага (nginx
  отдаёт дефолтный серт) там физически отсутствует, поэтому маршрутный nginx-vhost
  не пишется. Живая проверка `serving_ok` работает и там; LE-флоу в UI на Keenetic
  по-прежнему скрыт (HTTPS через KeenDNS). Изменённый `detour-cert` едет в
  `detour-keenetic_*.ipk` тем же манифестом. _Проверено на живом OpenWrt/nginx
  (home); на Keenetic путь не задействуется._

## [1.22.2] — 2026-06-29

Закрываем класс «зависаний opkg по IPv6» на Keenetic автоматически (по фидбэку с
железа: каждая установка вешалась на резолве `raw.githubusercontent.com` по IPv6,
пока вручную не прописывался `prefer_family = IPv4` в `/opt/etc/wgetrc`).

### Исправлено

- **opkg/wget на Keenetic больше не виснет на IPv6 к GitHub.** В РФ IPv6-маршрут к
  `raw.githubusercontent.com` блокируется/таймаутится, а у opkg нет флагов
  таймаута — один зависший `wget` держал `/opt/tmp/opkg.lock` десятками минут и
  морозил всю установку. Теперь `prefer_family = IPv4` (+`timeout`/`tries`)
  прописывается в `/opt/etc/wgetrc` **автоматически**: postinst панели сеет его при
  установке, а `detour-update` (`ensure_ipv4_wget` в `ensure_feed`) дописывает его
  перед любой opkg/wget-операцией — так что и уже стоящие коробки чинятся на
  следующей проверке. Идемпотентно (не дублирует, если строка уже есть). _Keenetic._
- `entware-bootstrap.sh` переведён с `inet4_only` на проверенный на устройстве
  `prefer_family = IPv4`.

## [1.22.1] — 2026-06-29

Багфиксы по фидбэку с живого Keenetic (первая фид-установка вскрыла настоящие
проблемы, а не только переход).

### Исправлено

- **Updater врал об успехе при провале установки.** opkg в Entware возвращает код
  выхода `0`, даже когда `opkg install` падает на неудовлетворённой зависимости —
  пакет молча остаётся старой версии. Из-за этого панель 1.22.0 «установилась»
  (статус `applied`), а на деле осталась 1.21.0. Теперь `cmd_apply`/`cmd_apply_local`
  **проверяют, что версия реально поднялась** (через `/opt/etc/detour/version` и
  разбор версии из самого `.ipk`), иначе пишут `error`, а не `applied`. _Обе
  платформы._
- **Keenetic не мог скачать наш HTTPS-фид через opkg.** Дефолтный Entware
  `wget-nossl` не умеет HTTPS вообще, а `wget-ssl` виснет на IPv6 к
  заблокированному в РФ `raw.githubusercontent.com`. Теперь на Keenetic
  `ensure_singbox`/`ensure_tpws` и кнопки «Обновить sing-box/zapret» качают `.ipk`
  **через `curl -4`** (новый `feed_curl_install`) и ставят локально, минуя
  сломанный загрузчик opkg. Это и есть рабочий обходной путь, найденный владельцем.
- **Чип sing-box на Keenetic показывал `0.0.0 · 0.0.0` и ложный амбер «в фиде
  нет».** До миграции наш пакет `sing-box` не установлен (стоит Entware
  `sing-box-go`), а `opkg list-upgradable` его не видит. Теперь `bins-check`/
  `tpws-check` на Keenetic читают доступную версию **прямо из фида (`Packages` по
  curl)** и показывают «доступна: 1.13.14 — есть обновление», а кнопка ставит её.
  «Новее на GitHub» теперь сравнивается с версией **фида**, а не установленной —
  иначе амбер горел вечно.

### Изменено

- **`entware-bootstrap.sh` / `deploy_keenetic.py` устойчивее к РФ-блокировкам.**
  bootstrap ставит `wget-ssl`, пишет `inet4_only` в `/opt/etc/wgetrc` и тянет
  бинарники `curl -4`; deploy грузит локально собранные `.ipk` фида на роутер и
  ставит их без скачивания на устройстве (самый надёжный путь).
- **Панель отдаётся с `Cache-Control: no-cache` (lighttpd).** После апдейта
  редизайн UI больше не «залипает» в кэше браузера до Ctrl+F5.

## [1.22.0] — 2026-06-28

### Добавлено

- **Свой mipsel opkg-фид для Keenetic — sing-box и tpws теперь свежие.** Раньше на
  Keenetic sing-box приходил из Entware-пакета `sing-box-go`, который ведут
  мейнтейнеры Entware и который сильно отстаёт (на устройстве было `1.13.3-2`,
  когда вышла `1.13.14`). Теперь `build_feed.py --arch mipsel` собирает наш
  **собственный фид** `feed/mipsel` из upstream-релизов: sing-box —
  ассет `-linux-mipsle-softfloat-musl` (32-бит LE MIPS, soft-float, полностью
  статический musl — исключает `Error relocating`), tpws — `binaries/linux-mipsel`
  из bol-van/zapret. Панель ставит их через `Depends: sing-box, tpws-zapret`, как
  на OpenWrt, и обновляет кнопками в шапке. Entware `sing-box-go` остаётся
  фолбэком для `Depends: sing-box`, если наш фид недоступен; `ensure_singbox`
  мигрирует на наш пакет и снимает `sing-box-go`. _Keenetic (паритет с OpenWrt)._
- **Авто-публикация фида расширена на mipsel.** `feed_autopublish.py` и
  CI-воркфлоу теперь публикуют ОБА фида — `aarch64` (OpenWrt, с nfqws2) и `mipsel`
  (Keenetic, sing-box + tpws). `publish_feed` стал мульти-арх-безопасным: подтягивает
  текущую ветку `feed` и сохраняет соседнюю арх-папку (публикация mipsel не сносит
  aarch64, и наоборот), оставаясь одним squash-коммитом.

### Изменено

- **Keenetic-пакет панели больше не несёт бинарников.** Bundled-`tpws` убран —
  оба бинарника (sing-box + tpws) теперь из фида. `deploy_keenetic.py` и
  `entware-bootstrap.sh` прописывают mipsel-фид и ставят оба пакета ДО установки
  панели. UI обновления sing-box/tpws на Keenetic переключён на единый
  фид-флоу (как OpenWrt); амбер-чип «вышла новее на GitHub» снова активен на
  Keenetic (мы теперь владеем фидом и можем бампнуть). Старый action
  `singbox_opkg_upgrade` оставлен алиасом на `bins-apply` для окна апгрейда.

> ⚠ **Не проверено на живом Keenetic-железе.** Сборка фида и `.ipk` (ELF, раскладка
> пакета, пути `/opt`) проверены локально, мульти-арх-публикация проверена на
> локальном фейк-remote. Но поведение на устройстве — что opkg ставит наши
> `Architecture: all` mipsel-пакеты, sing-box реально стартует, миграция с
> `sing-box-go`, fail-closed рестарт — **требует валидации удалённым владельцем
> Keenetic**. **Первая** установка фид-панели должна добавить mipsel-фид ДО
> `opkg install` (это делают `deploy_keenetic.py` / `entware-bootstrap.sh`); кнопка
> самообновления в панели НЕ перекидывает с дофидовой версии на фидовую.
>
> ⚠ **zapret2 (nfqws2) для Keenetic невозможен** и НЕ портирован: KeeneticOS не
> предоставляет NFQUEUE — движок физически не запустится. Это единственная
> легальная причина не делать паритет; в UI zapret2 на Keenetic уже задизейблен.

## [1.20.1] — 2026-06-28

### Исправлено

- **Обновление бинарника sing-box больше не открывает окно «трафик напрямую».**
  Раньше при `opkg upgrade sing-box` (чип версии → «Обновить sing-box») prerm
  фид-пакета делал `/etc/init.d/sing-box stop` **без** fail-closed-флага — это
  полный teardown: снимались все REDIRECT-правила, уничтожались ipset'ы,
  рестартился dnsmasq. На время подмены бинарника (до `restart` из postinst)
  трафик, который обычно идёт через VPN, **тёк напрямую** (в режиме «всё через
  VPN» — практически весь). Это нарушало инвариант «прямой трафик — только при
  РУЧНОЙ остановке sing-box». Теперь окно закрыто на трёх уровнях:
  - `detour-update bins-apply` держит флаг `/tmp/.singbox-keepfw` на всё время
    `opkg upgrade` (со стороны панели — действует сразу, независимо от версии
    установленного фид-пакета);
  - prerm фид-пакета (`build_feed.py`) тоже выставляет флаг перед `stop` — на
    случай ручного `opkg upgrade sing-box` по SSH мимо панели;
  - `stop_service` при выставленном флаге **оставляет REDIRECT-правила на месте**
    (зазор демона фейлит CLOSED: LAN → :12345 → connection refused, не напрямую),
    а флаг старше 2 мин считает «осиротевшим» (прерванный `opkg`) и всё-таки
    сносит правила — чтобы ручной «Выкл» нельзя было заблокировать застрявшим
    флагом.

  Рестарт и смена VPN-профиля/цепочки и раньше были fail-closed — затронуто
  только обновление бинарника. _Keenetic уже не течёт здесь_ (sing-box —
  Entware `sing-box-go`, чей пакет не трогает iptables detour, а init держит
  marker через `singbox_opkg_upgrade`) — правка не требовалась. ⚠️ tpws-zapret и
  nfqws2 при обновлении бинарника по-прежнему кратко снимают свой
  DPI-bypass-REDIRECT (это не VPN и не утечка IP — отдельная семантика); их
  fail-closed-паритет — отдельной задачей.

## [1.20.0] — 2026-06-27

### Добавлено

- **Настраиваемый объём файла для замера скорости VPN.** Раньше замер всегда качал
  фиксированные ~8 МБ — на быстрых каналах это упиралось в потолок ~50–60 Мбит/с
  (маленький файл скачивается за доли секунды, и среднее съедают TLS-рукопожатие и
  разгон TCP, не давая выйти на полку). Теперь в «Статусы VPN» рядом с тумблером
  «Замер скорости активного» появился селектор **«Объём замера»** с пресетами:
  - **8 МБ — эконом (как сейчас)** — таймаут 5 с, трафик ≤ 8 МБ;
  - **30 МБ — стандарт** — ≈ до 150 Мбит/с, таймаут 8 с;
  - **75 МБ — точный** — ≈ до 350 Мбит/с, таймаут 12 с;
  - **150 МБ — максимум** — ≈ до 600 Мбит/с, таймаут 15 с.

  Чем больше файл — тем точнее замер на быстром канале, но тем больше реального
  трафика через VPN за один замер (расход ограничен `min(объём, скорость × таймаут)`).
  Таймаут curl масштабируется под объём автоматически. Настройка хранится в
  `settings.health_speed_bytes`; `detour-health` строит из неё URL Cloudflare
  (`__down?bytes=…`) и таймаут. _Обе платформы_ (файлы общие, шим не затронут; на
  Keenetic так же — при наличии curl в Entware, как и прежде).

## [1.19.2] — 2026-06-27

### Исправлено

- **sing-box больше не забивает tmpfs логом и не валит панель (критично для
  Keenetic).** `render_chain_config()` генерировал конфиг с уровнем лога
  `"level": "info"` — sing-box пишет строку на каждое соединение (~40-50 МБ/сутки).
  На Keenetic `/var/log` — это tmpfs (~124 МБ всего): за ~3 дня аптайма лог
  заполнял весь `/tmp` на 100% и ронял всё, что пишет во временную ФС — обновление
  панели (`curl: (23) Failure writing output`), whitelist/сессии в UI
  (`lighttpd … No space left on device` → ложный `401`), health-check и замер
  скорости (`detour-health.db` не создавался). Лог был **единственным** потребителем
  tmpfs — поэтому одно исправление лечит весь каскад. Сделано:
  - уровень лога `info` → **`warn`** (≈100× меньше записей); _обе платформы_;
  - вывод лога теперь идёт в **`$SINGBOX_LOG`**, а не в хардкод `/var/log/sing-box.log`.
    На Keenetic это `/opt/var/log/sing-box.log` (персистентный USB, **вне tmpfs**) —
    тот самый путь, который уже ожидали просмотрщик логов в панели и
    `detour-logbridge`; раньше конфиг писал мимо них в tmpfs. На OpenWrt путь не
    меняется (`/var/log`, обрезается init-скриптом при каждом старте);
  - Keenetic-init `S52detour-singbox` теперь **обрезает лог при старте** (паритет
    с OpenWrt-init `> $LOG`) — обрезка на месте, чтобы `tail -f` логбриджа выжил.

  ⚠️ На уже установленных роутерах фикс применяется при следующей пересборке
  конфига (смена профиля/режима/рестарт sing-box) — старый `config.json`
  продолжит писать `info` в tmpfs до регенерации.

- **`detour-update`: моментальный выход при `curl (23)` вместо 4 бесполезных
  ретраев.** Код 23 = «не могу записать вывод» (переполнен `/tmp`) — повтор не
  поможет; теперь сразу понятная ошибка `cannot write output (disk full?)`.
  _Обе платформы._

- **`subscription-refresh`: понятная ошибка при HTTP 5xx / HTML вместо подписки.**
  Раньше при ответе сервера `502 Bad Gateway` (nginx-страница) парсер отвергал все
  «строки» HTML и отдавал невнятное `no profiles parsed (skipped=7)`. Теперь curl
  снимает HTTP-статус (`-w %{http_code}`): при коде ≥400 — `server returned HTTP
  502, not a subscription feed`, при HTML-теле — `server returned an HTML page,
  not a subscription feed`, при сетевом сбое — явное сообщение. _Обе платформы._
  (Проблема подписки Varyen из баг-репорта — на стороне провайдера, 502; код
  Detour ни при чём, фикс лишь делает диагноз читаемым.)

### Исправлено

- **Push не работал в Safari на macOS (обе платформы).** При включении тумблера
  push (или попытке отправить тестовый) в Safari не появлялось окно запроса
  разрешения, и подписка не создавалась. Причина — `pushEnable()` вызывал
  `Notification.requestPermission()` уже ПОСЛЕ `await api("push_config")`. Safari
  (macOS/iOS) показывает диалог разрешения только если `requestPermission()`
  вызван синхронно внутри пользовательского жеста (transient activation); любой
  `await` до него «съедает» активацию, и Safari молча возвращает `default` без
  диалога (Chrome/Firefox это прощают — потому баг проявлялся только в Safari).
  Теперь разрешение запрашивается ПЕРВЫМ действием, до любого сетевого запроса.
  Бэкенд (`detour-push`, VAPID/ES256) не менялся — он уже корректен для Apple
  Web Push (`aud` берётся из origin эндпоинта). Чистая правка фронтенда в общей
  `index.html` → чинит обе платформы.

- **Keenetic: не было живого лога при обновлении самой панели.** На OpenWrt лог
  установки стримится вживую (`apply_log`), а на Keenetic был пуст — потому что
  `apply_log` отдаёт та же самая `lighttpd` (`S51detour-panel`), которую установка
  панели гасит в `prerm` (`S51detour-panel stop`). Пока шла установка, веб-сервер
  с логом был недоступен, и панель ничего не показывала (а `pollPanelUpdate`
  перезагружал страницу раньше, чем лог успевал отрисоваться). Теперь
  `panel_update_apply`/`panel_update_local` на Keenetic ставят маркер
  `/tmp/detour-panel-selfupdate`, по которому `prerm` НЕ останавливает панель
  (CGI/HTML перечитываются на каждый запрос — отдавать со старого процесса во
  время подмены файлов безопасно), а `postinst` перезапускает её одним `restart`
  в самом конце и снимает маркер. `lighttpd` остаётся доступной почти весь
  апдейт → лог `apply_log` стримится вживую, как на OpenWrt. Фронтенд
  (`startPanelUpdatePolling`) теперь на обеих платформах ведёт завершение/

  > ⚠️ **Срабатывает со СЛЕДУЮЩЕГО апдейта.** Первый апдейт на сборку с этим
  > фиксом всё ещё выполняет СТАРЫЙ `prerm` уже установленной версии (он гасит
  > панель), поэтому лог при нём появится только после возврата панели (полный,
  > пост-фактум) — это штатно обрабатывается фолбэком. Каждое последующее
  > обновление панели стримит лог вживую.
  >
  > Обновление бинарников (sing-box из Entware) на Keenetic панель НЕ
  > перезапускает и стримит лог и так — затронут только апдейт самой панели.

### Безопасность

- **Защита от подбора пароля панели (обе платформы).** Действие `login` теперь
  считает неудачные попытки **по IP источника** (за прокси-на-роутере — по первому
  hop'у `X-Forwarded-For`, т.к. локальный реверс-прокси доверенный). После 8
  неудач за 15-минутное окно вход блокируется с эскалацией: 30 с → 2 мин → 10 мин
  → 30 мин (ответ `HTTP 429` + `Retry-After`, пароль при блокировке вообще не

- Новые файлы: `/usr/sbin/detour-push` (VAPID-отправитель), `/www/detour/sw.js`

## [1.10.0] — 2026-06-13

### Добавлено

- **Функциональная проверка работоспособности VPN — реальный запрос через
  sing-box, раз в час.** Пинг (`detour-ping`) отвечает лишь «сервер доступен»,
  но не «трафик реально идёт»: VPN может пинговаться зелёным, а YouTube / нужный
  сайт на нём не открывается (заблокирован/сломан на выходе). Новый
  `/usr/sbin/detour-health` поднимает **отдельный, временный** экземпляр sing-box
  (свой clash-API контроллер на `127.0.0.1`, без inbound'ов, **не трогает** боевой
  :12345), загружает в него outbound'ы **всех** профилей и через clash-API
  `/proxies/<tag>/delay` делает **настоящий HTTP-запрос каждой целевой ссылки
  через каждый VPN**. Профиль «рабочий», только если открылись ВСЕ цели. Цели по
  умолчанию: `youtube.com/generate_204`, `redirector.googlevideo.com/generate_204`
  (видео-CDN — ловит «главная грузится, видео нет») и `google.com/generate_204`;
  список редактируется в панели. Запускается из cron (OpenWrt, минута 41) и из
  `detour-cron` (Keenetic, раз в час); один прогон ~26 c на 114 профилей.
- **Бейдж «Работает» в панели** — рядом с пингом в выпадающем списке профилей,
  отдельная колонка в «Статусы VPN» и строка в карточке активного VPN. Подсказка
  показывает, какие именно цели прошли/упали (`YouTube ✓ · Google ✗`). На странице
  «Статусы VPN» — кнопка «Проверить работу всех», тумблеры и редактор целей;
  у каждой строки кнопка ⚕ для мгновенной перепроверки одного VPN.
- **Опциональное авто-переключение.** Если активный профиль (одиночный, не
  цепочка) провалил функциональную проверку — `detour-health` подтверждает сбой
  повторной пробой и переключается на ближайший рабочий профиль через тот же
  pipeline активации, что и панель (внутренний вызов `detour-api activate`).
  Выключено по умолчанию; включается тумблером. Событие пишется в
  `/tmp/detour-health.switch` и показывается в панели.

### Изменено

- CGI `detour-api`: новые действия `health_status` / `health_check` (полный
  фоновый прогон или быстрая проверка одного `id`) / `health_urls` (GET/POST
  списка целей) / `health_config` (тумблеры без рестарта sing-box). Флаги
  `health_check_enabled` / `health_auto_switch` хранятся в `settings.json` и
  сохраняются при переписывании настроек (как `vpn_redirect_ifaces`).

### Технические детали

- Все outbound'ы грузятся в ОДИН временный sing-box (ленивый дозвон — дёшево);
  если конфиг отвергнут из-за битого outbound, список профилей делится пополам
  рекурсивно, пока битый не изолируется и не пометится «down» — без медленного
  перебора по одному. Каждая цель пробуется с одним ретраем (гасит редкие
  transient-ошибки выхода, как ICMP-ретрай в `detour-ping`).
- Требует sing-box, собранный с `with_clash_api` (наш фид-бинарник — да). Если
  clash-API недоступен (напр. Entware `sing-box-go` на Keenetic) — `detour-health`
  это определяет проверочной пробой, ставит маркер `/tmp/detour-health.unsupported`
  и тихо выходит (панель прячет UI), а не помечает все VPN сломанными.
- Временный sing-box слушает clash на **по-прогонном** порту (`19390 + PID%1000`),
  НЕ на фиксированном: иначе экземпляр, осиротевший после убитого сигналом прогона
  (SIGHUP от отсоединённой/завершённой сессии), занимал бы порт, и следующий прогон
  цеплялся бы к мёртвому инстансу — все пробы в таймаут, ложное «всё не работает».
  Плюс sweep подчищает осиротевшие инстансы (`reap_orphans`) и ловит HUP в trap.
- Фоновый sweep из панели запускается через `start-stop-daemon -S -b` (на роутере
  нет `setsid`/`nohup`): голый `&` получил бы SIGHUP при завершении CGI и умер бы
  на полпути. В отличие от дешёвого ICMP-пинга, проба = полный TLS/Reality-хендшейк
  на профиль, поэтому очередь идёт аккуратным ручейком: не более `MAXJOBS=4`
  одновременных проб с паузой `PACE=1`с между батчами (чтобы не залпом ломиться во
  все соединения и не мешать живому VPN), а мёртвые серверы коротко замыкаются
  (профиль, проваливший первую цель, не тратит ретраи на остальные). Полный sweep
  114 профилей — ~4-5 мин раз в час; точечная перепроверка одного VPN — секунды.
- Пробы ждут через `wait $pids` (только PID'ы проб), НЕ через голый `wait`: временный
  sing-box — фоновый потомок шелла, и `wait` без аргументов завис бы на нём навсегда
  (это сервер, он не завершается) → sweep/one-mode зависал после проб, не доходил до
  `stop_sb`, инстанс утекал, а перепроверки по кнопке штабелировались. Это и была
  причина «проверка долго идёт и возвращает пустой ответ».
- Колонка «Проверен» в «Статусы VPN» — две строки: когда последний раз мерился пинг
  и когда последний раз гонялась функц. проверка. Кнопка ⚕ крутится-спиннером на
  время проверки (как кнопка пинга).

## [1.2.0] — 2026-06-01

### Порт на Keenetic / Entware (KeeneticOS, mipsel)

Первый рабочий порт всего стека на **KeeneticOS + Entware** (MT7621, mipsel),
проверен на реальном железе (sing-box 1.13.3, ABI в порядке).

- **Slim-пакет.** `detour-keenetic` больше не бандлит sing-box — он ставится из
  фида Entware по `Depends: sing-box` (пакет `sing-box-go` → `/opt/bin/sing-box`,
  ABI гарантирован сборкой). Размер пакета упал с **22.5 МБ до ~200 КБ**; в пакете
  остаётся только `tpws` (zapret в фиде Entware нет).
- **Обновление sing-box из OPKG по кнопке** — `opkg update && opkg upgrade
  sing-box-go`, отключение собственного автозапуска пакета и перезапуск сервиса
  (только Keenetic).
- **`all-except` маршрутизация** в firewall-хуке Keenetic: REDIRECT всего LAN TCP
  на sing-box, кроме приватных/loopback/CGNAT, апстрим-сервера и whitelist.
- **Подписки** портированы: встроенный pure-Lua `cjson.safe` (C-шного `lua-cjson`
  в фиде нет; проверен против системного lua-cjson) + шим путей под `/opt`.
- Панель отдаётся на `/detour/`; `detour-api` ремапит все пути под `/opt` на
  Entware (на OpenWrt — инертно).

### Изменено (обе платформы)

- **Подпись `.ipk` опциональна** — обновление панели и `detour-bins` больше не
  требует `.ipk.sig`. С подписью — проверка + бэкап, без неё — установка напрямую
  через opkg. Это же включает обновление панели на Keenetic.
- **Одна кнопка Старт/Стоп** для sing-box и zapret (переключается по состоянию).
- **«Kill Switch» → «Все через VPN»** — понятнее.
- **Логотип в шапке** заменён на favicon Detour.
- `status` теперь отдаёт поле `platform` (`openwrt` | `keenetic`).

### Исправлено

- **PATH в CGI/хуках/init.d** — lighttpd и NDM дают минимальный PATH, а инструменты
  Entware лежат в `/opt/bin:/opt/sbin`. Без экспорта PATH `sed`/`openssl`/`iptables`
  молча не находились → ложные `auth`-401 и отсутствие правил файрвола.
- Зависимости пакета: убраны `start-stop-daemon` (апплет busybox, не отдельный
  пакет) и `lua-cjson` (нет в фиде), из-за которых opkg отказывался ставить пакет.

## [1.0.0] — 2026-05-30

Первый публичный релиз Detour — веб-панель управления обходом блокировок для
роутеров GL.iNet / OpenWrt.

### Добавлено

- **Веб-панель (SPA)** на `/www/detour/` (uhttpd, порт 8080): режимы
  маршрутизации, редакторы списков доменов, управление профилями прокси, статус
  сервисов и обновления.
- **Shell CGI API** (`/www/cgi-bin/detour-api`) — бэкенд панели на BusyBox ash
  и Lua, без внешних рантаймов на роутере.
- **sing-box** (порт 12345) — прозрачный прокси Trojan/VLESS для маршрутизации
  выбранных доменов и подсетей через зарубежный сервер (режим redirect).
- **zapret-tpws** (порт 1081) — DPI-bypass прозрачный прокси без внешнего
  сервера.
- **Маршрутизация** через dnsmasq + ipset и iptables nat PREROUTING:
  zapret-домены → `:1081`, домены под режимом sing-box → `:12345`, остальное —
  напрямую.
- **Самообновление**: подписанные usign `.ipk`, установка через opkg, проверка
  GitHub Releases по cron (раз в 6 ч) с плашкой в шапке панели.
- **Двухпакетная схема релиза:** slim-панель `detour` (~90 КБ) и отдельно
  версионируемые бинарники `detour-bins` (`sing-box` + `tpws-zapret`, ~22 МБ),
  плюс оффлайн-установщик `detour-full-vX.Y.Z.tar.gz`.
- **Утилита `detour-update`** на роутере: `check` / `apply` / `bins-apply` /
  `rollback` / `status` / `selftest` и установка из локальных файлов.
- **Инструменты на рабочей машине:** `deploy_router.py` (унифицированный деплой
  по SSH с автоопределением особенностей устройства), `build_release.py`
  (сборка и публикация подписанных пакетов), `update_backups.py` (снятие
  состояния роутера), `usign_compat.py` (usign-подпись на Python).
- **Поддержка парка из нескольких роутеров** через `routers.local.json`
  (референс: GL-BE9300 на nftables fw4 и GL-MT6000 на iptables fw3).
- **Защитные механизмы платформы:** отключение сломанного MPTCP
  (`net.mptcp.enabled=0`), открытие портов через nftables `fw4`, hotplug-guard
  для восстановления правил после смены интерфейсов.
